# Методические указания преподавателям по оцениванию

## Дисциплина: Модели и методы интеллектуального поиска

### 1. Общие положения

Настоящие методические указания предназначены для преподавателей, реализующих дисциплину «Модели и методы интеллектуального поиска» в рамках профессиональной роли **ML Researcher**. Дисциплина состоит из 5 модулей и направлена на формирование компетенций в области разработки, настройки и оценки интеллектуальных поисковых систем с использованием методов машинного обучения, обработки естественного языка, векторного поиска и нейросетевых подходов.

### 4. Организация контроля

#### 4.1 Формы контроля

| Тип контроля | Форма | Описание |
| :--- | :--- | :--- |
| **Текущий** | Устный опрос, мини-тест, практическое задание, эссе | Проводится на лекционных и семинарских занятиях |
| **Промежуточный** | Контрольная работа, лабораторные работы | Проводится после завершения модулей 1–4 |
| **Итоговый** | Экзамен, защита проектов, итоговый тест | Проводится по завершении дисциплины |

#### 4.2 Рекомендации по проведению оценочных мероприятий

**4.2.1 Текущий контроль**

- **Устный опрос** рекомендуется проводить в форме фронтального опроса или индивидуального собеседования. Вопросы должны проверять не только знание фактов, но и понимание причинно-следственных связей. Рекомендуемое время на ответ — 2–3 минуты на вопрос.

- **Мини-тесты** рекомендуется проводить в начале занятия (5–10 минут) для проверки усвоения материала предыдущей лекции. Используйте закрытые вопросы (выбор одного правильного ответа). Время на выполнение — 5–10 минут.

- **Практические задания** требуют от студентов применения знаний для решения конкретной задачи. Оценивайте не только правильность ответа, но и обоснованность выбора методов, полноту анализа и качество аргументации.

- **Эссе-миниатюры** — эффективная форма проверки аналитических способностей. Оценивайте: глубину понимания проблемы (40%), обоснованность предложенного решения (40%), чёткость и лаконичность изложения (20%).

**4.2.2 Лабораторные работы**

Каждая лабораторная работа сопровождается вопросами к защите. При оценке руководствуйтесь следующими принципами:

1. **Код должен быть воспроизводимым.** Проверьте наличие всех необходимых импортов, комментариев и документации.
2. **Метрики должны быть обоснованы.** Студент должен объяснить выбор метрик для конкретной задачи.
3. **Анализ ошибок — ключевой элемент.** Отличник должен уметь указать, где его модель «ломается» и предложить пути улучшения.
4. **Визуализации должны быть интерпретируемы.** Графики должны иметь подписи осей, легенды и заголовки.
5. **Выводы должны следовать из результатов.** Не принимайте выводы, которые не подтверждаются полученными данными.

**Рекомендуемый порядок защиты лабораторных работ:**
1. Демонстрация работающего кода (5 минут).
2. Презентация результатов с обоснованием выбора методов (5 минут).
3. Ответы на вопросы преподавателя (5 минут).

**4.2.3 Контрольная работа**

Рекомендуемая структура контрольной работы:

| Тип задания | Количество | Баллы (max) |
| :--- | :---: | :---: |
| Архитектуры и модели | 1–2 | 20–30% |
| Расчёт метрик | 1 | 20% |
| Понимание запроса | 1 | 25–30% |
| Генеративное переформулирование | 1 | 25–30% |

При проверке задач на метрики обязательно проверяйте:
- Правильность определения идеального порядка (для NDCG).
- Корректность использования логарифмического дисконта.
- Арифметические вычисления (допустимы небольшие расхождения при округлении).

**4.2.4 Итоговый экзамен**

Рекомендуемая структура экзаменационного билета:

| Компонент | Время | Баллы (max) |
| :--- | :---: | :---: |
| Теоретический вопрос | 20 мин | 50% |
| Практическая задача | 30 мин | 50% |

**Теоретические вопросы** должны охватывать ключевые темы модулей. При оценке обращайте внимание на:
- Полноту раскрытия вопроса.
- Глубину понимания (способность объяснять механизмы, а не просто перечислять факты).
- Умение приводить примеры и связывать теорию с практикой.

**Практические задачи** могут включать:
1. Расчёт метрик качества ранжирования.
2. Проектирование архитектуры поисковой системы для заданного домена.
3. Написание псевдокода для fine-tuning BERT.
4. Анализ ошибок ранжировщика с предложением способов устранения.

### 6. Рекомендации по работе с модулями

#### 6.1 Модуль 1. Введение в интеллектуальный поиск

**Лекция:** Уделите внимание эволюции подходов к поиску — от булевых моделей до семантического поиска. Обязательно объясните, как инвертированный индекс ускоряет поиск, и почему он строится до этапа ранжирования.

**Лабораторная работа (Cranfield):** Обратите внимание на правильность реализации BM25 (параметры k1 и b) и TF-IDF. При оценке аналитической части проверьте, как студенты интерпретируют разницу между моделями на специфичных запросах.

**Деловая игра:** Следите, чтобы команды обосновывали выбор методов не интуитивно, а на основе математических свойств. На этапе защиты будьте строги — требуйте аргументации, основанной на метриках (Recall, AUPRC), а не на Accuracy.

#### 6.2 Модуль 2. Машинное обучение в поиске

**Лекция:** Детально разберите различия между pointwise, pairwise и listwise подходами. Используйте примеры из реальных систем. Объясните, почему LambdaMART оптимизирует NDCG напрямую.

**Лабораторная работа (LTR):** Следите за правильностью группировки документов по query_id — это частая ошибка. При оценке hyperparameter tuning проверяйте, что студенты понимают влияние глубины деревьев и количества листьев на качество ранжирования.

**Задание «Слепая дегустация»:** Убедитесь, что студенты понимают разницу между Bi-Encoder и Cross-Encoder не только на уровне архитектуры, но и на уровне сценариев применения (candidate generation vs reranking).

#### 6.3 Модуль 3. Обработка естественного языка для поисковых систем

**Лекция:** Акцентируйте внимание на том, что стоп-слова могут быть критичны в некоторых доменах (медицина, юриспруденция). Используйте примеры, где удаление стоп-слов меняет смысл фразы.

**Лабораторная работа (NLP):** Проверяйте качество очистки текста. Ошибки на этом этапе могут свести на нет всю кластеризацию. При оценке кластеров смотрите на интерпретируемость топ-слов — студенты должны уметь объяснить, почему документы попали в тот или иной кластер.

**Анализ ошибок:** Требуйте не просто перечисления ошибок, а объяснения причин (полисемия, шум в тексте, неоптимальный выбор k).

#### 6.4 Модуль 4. Глубокое обучение и нейросетевые подходы

**Лекция:** Объясните Self-Attention визуально — используйте схемы для иллюстрации того, как каждое слово взаимодействует с каждым. Обязательно разберите trade-off между Bi-Encoder и Cross-Encoder.

**Лабораторные работы:** Для успешного выполнения требуется доступ к GPU. Предоставьте студентам доступ к Colab Pro или GPU-серверам. При оценке fine-tuning проверяйте сходимость loss и финальные метрики.

**Контрольная работа:** В задаче на метрики проверяйте не только финальный ответ, но и промежуточные вычисления (DCG, IDCG). Допустимы небольшие расхождения при округлении, но идеальный порядок должен быть определён верно.

#### 6.5 Модуль 5. Специализированные виды поиска

**Лекция:** Используйте реальные примеры персонализации (YouTube, Netflix, Amazon) для иллюстрации. Обсудите этические проблемы: bias, информационную справедливость, конфиденциальность.

**Лабораторная работа:** Следите, чтобы студенты не просто строили модель, но и анализировали, для каких групп пользователей персонализация эффективна, а для каких — нет. Это ключевой аспект оценки.

### 7. Рекомендации по формированию итоговой оценки

Итоговая оценка по дисциплине формируется на основе следующей формулы:

**Итог = 0.25 · Лекц. контроль + 0.40 · Лабораторные + 0.20 · Контрольная + 0.15 · Экзамен**

Где:
- **Лекц. контроль** — средневзвешенная оценка за все формы текущего контроля.
- **Лабораторные** — средняя оценка за лабораторные работы с учётом веса каждой.
- **Контрольная** — оценка за контрольную работу (по модулям 1–4).
- **Экзамен** — оценка за экзамен (теоретические вопросы + практические задачи + защита проектов).

**Шкала перевода баллов:**

| Оценка | Баллы (max 100) |
| :--- | :--- |
| Отлично | 86–100 |
| Хорошо | 70–85 |
| Удовлетворительно | 51–69 |
| Неудовлетворительно | 0–50 |

### 8. Типичные ошибки студентов и способы их предупреждения

| Типичная ошибка | Рекомендация преподавателю |
| :--- | :--- |
| Путает Bi-Encoder и Cross-Encoder | Используйте наглядные схемы, подчёркивайте разницу в обработке [query+doc] |
| Некорректно группирует данные по query_id для LTR | Проведите отдельный мини-воркшоп по работе с группами в LightGBM |
| Не учитывает Position Bias в поведенческих признаках | Акцентируйте внимание на проблеме self-fulfilling prophecy |
| Игнорирует лемматизацию в NLP-пайплайне | Требуйте фиксацию изменения размера словаря до/после очистки |
| Не анализирует ошибки кластеризации | В критериях оценивания выделите анализ ошибок как отдельный элемент |
| Путает Precision/Recall при расчёте метрик | Используйте матрицу путаницы для наглядного объяснения |
| Не обосновывает выбор гиперпараметров | Требуйте обоснование каждого выбранного параметра |
