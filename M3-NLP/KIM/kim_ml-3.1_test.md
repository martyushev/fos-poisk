- [ ] а) Стемминг.

### Задача 1. «Парадокс стоп-слова» (Архитектурное эссе-миниатюра)

**Контекст:** Вы строите медицинский поисковик для врачей. Ваш инженер предлагает агрессивно удалить все стоп-слова для экономии места в инвертированном индексе.

**Задание:** Напишите короткий аргументированный ответ инженеру (4–5 предложений). Объясните, почему фраза «отказ левой почки без признаков диабета» после удаления стоп-слов превратится в проблему для точности поиска. Укажите конкретный пример термина, который пострадает (например, предлоги в названиях препаратов или анатомических связях).

**Эталонный ответ:** Удаление предлогов «без» и «от» в медицинском контексте может инвертировать смысл: запрос «отказ почки без диабета» станет неотличим от «отказ почки диабет». Это приведет к выдаче нерелевантных карточек пациентов с диабетом. Предлог «без» является ключевым дифференцирующим признаком, аналогично предлогам в названиях препаратов (например, «раствор для инъекций» против «раствор для приема внутрь»).

### Задача 2. «Слепое пятно BERT» (Анализ граничных условий)

**Контекст:** Вы внедрили кросс-экодер BERT для переранжировки топ-10 выдачи интернет-магазина.

**Задание:** Опишите сценарий (паттерн запроса), при котором мощнейшая языковая модель выдаст абсолютно нерелевантный результат, уступающий простому BM25.

**Эталонный ответ:** Запрос "смартфон НЕ черный" или "зарядное устройство type-c длиной ровно 1.5 метра". BERT плохо справляется с точной арифметикой, отрицаниями и специфическими SKU, воспринимая числа как обычные токены, а не как числовые фильтры. BM25 отработает точнее на лексических совпадениях с артикулами.

### Задача 3. «Инвентаризация краулера» (Data Engineering)

**Контекст:** Вам нужно проиндексировать базу знаний крупной IT-компании. Она состоит из:

1. Внутренних Confluence-страниц (HTML + вложенные PDF).
2. Таблиц со списком серверов (JSON, поля: IP, owner, status).
3. Чатов техподдержки (неструктурированные потоки текста).

**Задание:** Распределите эти источники по типам (Неструктурированные / Полуструктурированные / Структурированные). Для каждого типа укажите главную сложность при создании единого поискового индекса.

**Эталонный ответ:**
- **Полуструктурированные:** Confluence-страницы. Сложность — извлечение чистого текста из HTML/PDF без потери структуры заголовков и таблиц.
- **Структурированные:** JSON-таблицы серверов. Сложность — интеграция точных атрибутов (IP, статус) в текстовый поиск без потери возможности фильтрации.
- **Неструктурированные:** Чаты техподдержки. Сложность — высокая зашумленность, сленг, обрывки диалогов и отсутствие явной тематики.
